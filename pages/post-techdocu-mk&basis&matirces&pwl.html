Code snippet: Basis matrix evaluation of piecewise linear

Introduction

In collocation, we often pre-compute basis matrices for evaluating Jacobian. This document gives some high-performance implementations. It will be updated by time.

Let's consider $N+1$ even-spaced node-value pairs $(x_i,y_i)$ where $i=1,\dots,N+1$ and $y_i=f(x_i)$. We require $x_i$ are distinct and ascendingly sorted. Assume $x_i\in[\underline{x},\bar{x}]$ be the domain of $x$, or in other word, the interval in which we want to interpolate $y$. Then, let's linearly project $x_i$ to $\tilde x_i\in[1,N+1]$ because a unit step length simplifies the computation. This operation can be done via $\tilde x = 1 + N(x-\underline{x})/(\bar{x} - \underline{x})$. Its reversion is $x=\underline{x}+(\tilde x-1)(\bar{x}-\underline{x})/N$.

Our purpose is to interpolate $(\tilde x_i, y_i)$ with function $\hat{f}(x):=\phi(x)\cdot c$.

Piecewise linear interpolation is easy, fast and its basis matrix is independent from sample values $y_i$. There are at most two non-zero elements (hat function) in $\phi$. So the performance bottle neck of evaluating $\phi$ of piecewise linear interpolation is locating non-zero hat functions in this $N+1$ length sparse vector $\phi$. In the worst case, this may require a $O(\log n)$ binary search. However, the projection of $x$ to $[1,N+1]$ makes it an $O(1)$ now.

Let's use $(I,V)$ format to define a sparse vector and $(I,J,V)$ format for a sparse matrix.

Let $\hat{x}\in[1,N+1]$ be a point to evaluate basis vector $\phi(\hat x)$. Let $lb:=\lfloor \hat{x} \rfloor$ and $ub:=\lceil \hat{x} \rceil$. Because $\hat{x}\in[1,N+1]$, its floor and ceil naturally locate non-zero elements in $\phi(\hat{x})$. Then, the algorithm is:

  1. If lb == ub

    1. true: $\hat x$ is exactly at a grid point. Let I=[lb,] and V=[1.0,] and return

    2. false: $\hat x$ is between two grid points $(lb,y_{lb})$ and $(ub,y_{ub})$. Thus I=[lb,ub] and V=[ub-x,x-lb] and return.

Linear extrapolation features require some modification of the above algorithm but the complexity is still $O(1)$.

When there are multiple $\hat{x}_i$, we need to construct a basis matrix $\Phi$. Easy to see that the time complexity is $O(n)$ which only depends on the number of $\hat x_i$ but independent from the degree of the interpolation.

The follows are example implementations:

using BenchmarkTools
using SparseArrays

"""
	mkbase_pwl(x::Real, N::Int)

Evaluates basis vector at point x in [1,N+1]. Returns an 2-vector Tuple
`(I,V,N+1)` for defining a sparse vector. Each vector has at most 2 elements.
Specifically, `Tuple{Vector{Int},Vector{Float64},Int}`. unsafe, no bound check
"""
function mkbase_pwl(x::Real, N::Int)
	lb = floor(Int, x); ub = ceil(Int, x)
    return (lb==ub) ? sparsevec([lb,],[1.0,],N+1) : sparsevec([lb,ub],[ub-x,x-lb],N+1)
end

"""
	mkbase_pwl(X::AbstractVector, N::Int)

Evaluates basis matrix at points in X in [1,N+1]. Returns a len(X)*(len(N)+1)
`SparseMatrixCSC{Float64, Int64}`. unsafe, no bound check
"""
function mkbase_pwl(X::AbstractVector, N::Int)
	Is = Int[]; Js = Int[]; Vs = Float64[]
	nr = length(X); nc = N+1;
	for i in 1:nr
		lb = floor(Int, X[i])
		ub = ceil(Int, X[i])
		if lb == ub
			push!(Is, i); push!(Js, lb); push!(Vs, 1.0)
		else
			push!(Is, i); push!(Is, i)
			push!(Js, lb); push!(Js, ub)
			push!(Vs, ub-X[i]); push!(Vs, X[i]-lb)
		end	
	end
	return sparse(Is,Js,Vs,nr,nc)
end

"""
	v2i(x::Real, N::Int, xmin::Real, xmax::Real)
	
Linearly projects x in [xmin,xmax] to range i in [1,N+1].
"""
function v2i(x::Real, N::Int, xmin::Real, xmax::Real)
	return 1.0 + N * (x-xmin) / (xmax - xmin)
end

"""
	i2v(i::Real, N::Int, xmin::Real, xmax::Real)
	
Linearly projects i in [1,N+1] to range x in [xmin,xmax].
"""
function i2v(i::Real, N::Int, xmin::Real, xmax::Real)
	return xmin + (i-1)*(xmax-xmin)/N
end

# Usage
@btime mkbase_pwl($23.3333, $1000);
@btime mkbase_pwl($23.3333, $10000);
@btime mkbase_pwl($23.3333, $100000);

The only 9 times allocations occur in creating the sparse vector instance. If we don't really need a sparse vector, we can degreade it to a tuple of $(I,V)$ vectors, this number can be 2 and it is even much faster than now.
