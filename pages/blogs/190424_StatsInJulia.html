Introduction¶

In another blog where I introduced how to prepare your Julia 1.0 for economic research, I roughly discussed the statistics/econometrics in Julia in the conclusion section. Though there is still a long way for Julia to go in econometrics, we can now do many fantastic jobs in statistics. This blog discusses how to do basic statistics in Julia 1.0. It depends on JuliaStats project and its packages. This is a project initiated by the Julia official. It is the base of the statistics in Julia language.

In Section 2, I provide a full list of the packages under JuliaStats project and some other packages. We will use these packages in this blog. In Section 3, I introduce how to use Julia do descriptive Statistics, e.g. histogram, QQ-plot. In Section 4, I introduce how to play with many statistical distributions in Julia. In Section 5, I introduce how to sampling on distributions or datasets. In Section 6, I introduce how to do hypothesis tests e.g. two-sample F tests. In Section 7, I introduce how to do basic regression analysis with MultivariateStats and GLM packages. In Section 8, I introduce how to do other common statistical analysis such as clustering and PCA. In Section 9, I discuss what other functions are required but not provided yet if we want to do more-specific econometric research, e.g. quantile regression and panel data models.

Making a table of descriptive statistics¶

When introducing a dataset in paper, authors are usually required to provide a summary table like:

Variable N min 25% quantile median mean std 75% quantile max
y 100 -1.00 -0.24 -0.20 -0.18 2.34 0.23 0.76
x 100 23.22 25.03 27.98 26.33 0.54 29.22 31.43

To make such a table, readers need to learn:

  1. how to compute basic statistics, e.g. sample size, mean and std, for a single variable
  2. how to construct or format such a table

For the two tasks, there may be some easy-APIs to automatically compute everything then do formatting. But we will finish the two jobs step by step, to get familiar with Julia's statistics.

Before Julia 0.7, common descriptive statistical methods, e.g. mean, are included in core Julia, which means users do not need to explicitly import/using a package to introduce them. But since v0.7, these functions are separated to a standard library StatsBase. (Yes, this package is one part of JuliaStats project) Now, we introduce this package then see several demos which show how to do basic statistics.

P.S. to explicitly display which pacakge a function belong to, we use import but not using to keep introduced packages private. But in practice, we usually using StatsBase to expose all exportable objects to current namespace to avoid re-writing "StatsBase" times by times

println("Please note the type of x: ",typeof(x), " which is a synonym of Vector{Float64,1}")

println("skewness: ", StatsBase.skewness(x) )
println("5th order sample central moment: ", StatsBase.moment(x,5) )
println("min  value & its location in x: ", findmin(x) )
println("max  value & its location in x: ", findmax(x) )
println("25% quantile: ", StatsBase.quantile(x, 0.25) )
println("75% quantile: ", StatsBase.quantile(x, 0.75) )

There are several things to note:

  1. Vector{T} is a simple presentation of one-way array Array{T,1}, where T is a wildcard denoting "type" in Julia type systems. In the same way, Matrix{T} is the simple presentation of two-way array Array{T,2}. And in Julia, Array{T,W} can be seen as the union set of Python's list and Matlab's array. They can do list operations also linear algebra operations.
  2. finding max/min in Julia does not use max() and min() but use findmax() and findmin(). The two functions return a 2-member Tuple whose 1st member is the max/min value and the 2nd is the location of max/min in the given iterable object. By the way, max() and min() are similar to those in C/C++. They just compare two arguments and return the larger/smaller one.

  1. using simple Array{T,2} structures then format it by oneself
  2. using DataFrame to make it look like a table

Plotting: histogram, QQ-plot, and density curve¶

In the section of descriptive statistics, authors usually provide illustrations about some important variables (e.g. dependent variable). This subsection discusses how to make common diagrams in Julia. Of course, using PyPlot API. Histogram & Density Curve¶

One of the most common diagram used in descriptive statistics is single-variable histogram. Sometimes, people also like to draw a histogram with a kernel-density curve. Actually, both histogram & kernel-density, as the most simple nonparametric estimations, need to be estimated. But here we do not discuss this topic because PyPlot automatically does this job unless we need some custom histogram (e.g. censored histogram). If you wanna learn how to make your custom colorful histograms, please refer to this page.

Here is a demo to draw histograms with an estimated kernel-density curve in Julia. You need the KernelDensity package of JuliaStats project.

In [ ]:
using PyPlot  # using PyPlot
    import KernelDensity  # under JuliaStats project
In [5]:
x = randn(2000) # sample data, a vector
kde_x = KernelDensity.kde(x) # univariate kernel estimation, default settings

figure()
    hist(x, 40, density = true); # 40 specifies how many sections to use, density = true displays density rather than frequency
    plot(sort(x),KernelDensity.pdf(kde_x,sort(x)),"--r") # sort x then use KernelDensity.pdf() to get the interpolated pdf value at x points
    # NOTE: sort x because plot() connect points according to their position in x
    xlabel("x"); ylabel("density"); grid(true);
    title(L"This is a latex title $N(\mu,\sigma)$");  # this is how to use latex string in PyPlot: L"string"

Readers are recommended to use NovDist.LatinHyperCube() in my another blog to better rebuild a distribution!

In this demo, I import KernelDensity rather than using it because I wanna let readers know which functions belong to KernelDensity. We can see KernelDensity.kde function estimates the kernel density of vector x with default settings (e.g. the bandwidth), like the density() function in R language. But please note, the returned object by KernelDensity.kde() is an interpolation-like object whose density value can be evaluated at a specific x0 via KernelDensity.pdf(kde_x,x0). Meanwhile, KernelDensity provides full functions for kernel estimation, users may read its documentations for more usage of this package.

By the way, readers are recommended to use NovDist.LatinHyperCube() in my another blog to quickly rebuild a distribution, esp. when they do Monte-Carlo based estimations.

QQ-plot¶

QQ-plot is an intuitive tool to diagnose if two given distributions are linearly correlated. If we wanna use QQ-plot to see if a given distribution $X$ follows another given distribution $F$, we need:

  1. experience quantiles $q_X(x)$ estimated from a given experience distribution $X$ (if $X$ is a sample set, computing $q_X(x)$ for $\forall x \in X$ via the rate of standard competition ranking on sample size)
  2. according to the experience quantiles, assuming $x$ is from a given distribution $F$; then for a specific $q_X(x_0)$, compute the $\tilde{x}_0$ following $F$ which lets $q_x(x_0) = q_F(\tilde{x}_0) $
  3. draw a curve $( \tilde{x}_0, x_0 )$ on x-y plate, this is QQ-line
  4. draw a reference curve $( \tilde{x}_0,\tilde{x}_0 )$ on the same plate

Because PyPlot does not provide quick functions for QQ-plots, we compute it here by ourselves (also for better understanding the QQ-plot). If you use other visualization tools, there may be some lazy functions to do this job.

And, the reason why we use the standard competition rankings ("1224") is that quantiles are defined on the standard competition ranking but not other kinds of rankings. Please read Wikipedia to learn the differences among ranking types.

Note: Actually, in Section 4, I will talk about the QQPair in Distributions package.

Here is a demo which tests if a sample vector x is from a $N(0,1)$ distribution:

In [3]:
using PyPlot # plot
    import StatsBase # lib under JuliaStats, we use it to compute rankings
    import Distributions  # lib under JuliaStats, for distribution types
In [9]:
x = randn(200)  # sample data, a given experience distribuion X to test

# compute experience quantiles of X
sort!(x) # sort x in-place; less allocations when x is large (well, that means millions-level vector)
expqt_x = StatsBase.competerank(x) ./ length(x) # compute COMPETITION rankings and divided by the sample size to get experience quantile of X

# get the corresponding x\tilde from N(0,1), the distribution we assume X to follow
# NOTE: use func.(x) to apply a scalar method on every element of array x and return a same-size array of results
theoqt_x = Distributions.quantile.( Distributions.Normal(), expqt_x )

# plotting
plot( theoqt_x, x, "." ); plot( theoqt_x, theoqt_x, "-" );
grid(true); xlabel("Theoretical quantiles of N(0,1)"); ylabel("Experience quantiles");

Distributions.jl and supported distributions¶

Distributions is a package under JuliaStats project. It provides supports for varies of defined statistical distributions: declaration, characters, properties, sampling, fitting, mixture model etc. It also allows developers to create their new statistical distributions in a standard work flow and integrate them with the current Distributions. Please refer to the documentation for more usage.

In R and Matlab, users may get used to methods like randn() and runif(), where distributions are not defined separately from relative operations. But this design is not philosophical good because it fuzzes actions and the objects that the actions work on. So, in ideal object-oriented (OO) designs, "distribution" objects are separated from operations like sampling. But in many full-OO designs, esp. some in python/Java, we may see a normal distribution class defined as:

# python 3 
class NormalDist:
    mu = 0.0
    sigma = 0.0
    def __init__(self, mu, sigma):
        self.mu = mu
        self.sigma = sigma
    def sample(self,N)
        ...

When we run NormalDist(0,1), we get an instance of "distribution" then can operate it as an object. It looks we have treat normal distribution as an object successfully. However, users may find a sample() method is defined inside NormalDist class, in which the method is actually an action to be applied on a normal distribution. This is another philosophical mistake happening often in many programs. I have to say it is so weired because it seems that a distribution can sample itself: in fact, it is we, the econometricians, who sample from the distribution.

Fortunately, we can see this problem is solved in Julia because of its multi-patches OO realization, where we use types rather than classes to do OO programming. One of the most significant difference (at least on the surface) is that functions are not allowed to be defined inside types (except initialization method). Distributions are defined as a identifying symbol (type) with defined parameter values. Functions decide their behaviors according to what type of a given distribution instance. It strictly distinguishes objects from operations. Though now we need to reload generic functions explicitly, we will not meet philosophical confusions like "someone pulls his/her hair to lift his/her own body", i.e. bootstrapping. (lol this is a cold joke)

Distributions follows such a philosophy separately defining the distributions to operate and many operations on these distributions. For example, we can define a normal distribution through d = Distributions.Normal(mu,sigma) then sample from it through Distributions.sample(d,200). Until now, this package has supported over 77 standard distributions:

Characteristics: pdf, cdf, inv-cdf, mgf¶

Generating random numbers from given distributions¶

Generating random numbers from a given distribution, in fact, may require different technologies, but fortunately, Distributions has reloaded Base.rand() for almost supported distributions. an easier way is just use rand(d,...):

In [103]:
d = Distributions.Beta(1.1,2.2)  # a Beta distribution instance
rndnums1 = rand(d, 500)  # just pass the defined distribution and sample size to Base.rand

d = Distributions.MvNormal( [1.0,2.0], [2.2 1.0; 1.0 1.5] )  # define a multivariate normal
rndnums2 = rand(d, 100)  # each column is a sample, totally 100 columns and 2 rows

hist(rndnums1, 40, density = true); println(size(rndnums2));

(2, 100)

However, when we define our own distribution type, we need to manually reload Base.rand() in some ways. There are some most common methods for different types of distributions to generate random numbers:

  1. for continuous distributions
    1. for univariate distributions & multivariate distributions without dimension-correlations
      1. the standard method is to compute inverse cdf function $x = F^{-1}(z)$ where $z$ is randomly generated from a (0,1) uniform distribution.
      2. using filtering-like methods when analytical pdf known
    2. for multivariate distributions with dimension correlations
      1. sampling on each dimension then approximate given correlations
      2. if copula applied, sampling on one dimensions then solve the other dimension
  2. for discrete distributions
    1. for univariate distributions & multivariate distributions without dimension-correlations
      1. project uniform (0,1) to probability space; i.e. concrete inverse cdf function
    2. for multivariate distributions with dimension correlations
      1. sampling without correlation first, then approximate the correlations

An inverse cdf function can be analytically defined or use interpolations to approximate (e.g. inv-cdf of normal distributions). If feasible, this method should be your first choice. Please consider using filtering-like methods only in the case of too-complicated distributions and small sample size.

To manually compute $F^{-1}(x)$ for case 1.A.a, the most common method, readers just need to do the following jobs if a distribution has been supported by Distributions:

In [86]:
d = Distributions.Beta( 1.5, 2.5 ) # define a beta distribution with two parameters

z = rand(5000); # sampling from (0,1) uniform, the prob space

rndnums = Distributions.quantile.( d, z )  # quanitle(), in fact, is equivalent to inverse cdf

hist(rndnums, 30, density = true)  # observe the distribution
xlabel("x"); ylabel("density"); title(L"\beta(0.5,0.5)");

If readers are interested in the exact forms of different distributions' inverse cdf functions, please go to another package StatsFuns. It concludes over one hundred common statistical functions e.g. many distributions' cdf/ccdf/logcdf/logccdf/inv-cdf and many high-accuracy irrational constants like $\sqrt{\pi}$. This package is quite useful when someone wanna do high-accuracy, atom-level, low-level, performance-required statistical computations.

Mixture models (look through)¶

One thing prior to this subsection is that Distributions only defines mixture models with generic functions (e.g. loglikelihood) but not provide estimation procedures. Readers are required to use other packages, e.g. GaussianMixtures, to do estimations. And, because Distributions provides loglikelihood for defined mixture models, readers who wanna use their own estimation programs then can easily combine the standard model types with their programs.

This is a general k-component mixture model: $$ f(x; \Theta, \pi) = \sum^K_{k=1} \pi_k f(x;\theta_k) $$ It is defined by k component distributions with parameter $\Theta = {\theta_1,\dots,\theta_K}$, and a prior distribution $\pi = {\pi_1,\dots,\pi_K}$ of each component. Therefore, we can naturally define such a mixture model via Distributions:

In [100]:
import Distributions

# define a mixture model composed by a normal distribution & a Beta distribution
# NOTE: pass components with parameters in a list, as the 1st par to MixtureModel
# NOTE: the 2nd par is a vector tells the prior probabilities of the two component distributions
# NOTE: if prior probs missed, using even prior distrbution (e.g. [0.25,0.25,0.25,0.25])
mixmod = Distributions.MixtureModel(
    [Distributions.Normal(1.0,2.0),
    Distributions.Beta(1.5,1.2)],
    [ 0.4,0.6 ]
)
println("Type of the 1st mixture model: ",typeof(mixmod))

# or we can define a mixture model with all the same distributions
# NOTE: the following sentence defines a 3-component GMM (gaussian mixture model)
mixmod = Distributions.MixtureModel(
    Distributions.Normal, # first, indicate what kind of distribution to mix
    [ (0.0,1.0), (1.2,2.5), (2.2,2.54) ], # parameter tuples in a collection
    [ 0.2,0.3,0.5 ]  # prior distribution
)
println("Type of the 2nd mixture model: ",typeof(mixmod))

# another method via map()
mixmod = Distributions.MixtureModel(
    map( mu -> Distributions.Normal(mu,1.2), [ 1.1,2.3,3.4 ] ), # using map() + annonymous function to generate similar components
    [ 0.2,0.3,0.5 ]  # prior distribution
)
println("Type of the 3rd mixture model: ",typeof(mixmod))

Type of the 1st mixture model: Distributions.MixtureModel{Distributions.Univariate,Distributions.Continuous,Distributions.Distribution{Distributions.Univariate,Distributions.Continuous}}
Type of the 2nd mixture model: Distributions.MixtureModel{Distributions.Univariate,Distributions.Continuous,Distributions.Normal}
Type of the 3rd mixture model: Distributions.MixtureModel{Distributions.Univariate,Distributions.Continuous,Distributions.Normal{Float64}}

As shown above, readers can intuitively and easily define their custom mixture models even in which different kinds of distributions are mixed.
